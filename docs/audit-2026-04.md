| `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` / `CLERK_SECRET_KEY` / `CLERK_WEBHOOK_SECRET` | Clerk (현재 직접 사용 X, 웹훅만 호환) |

### 5.3 관찰 사항

- **`NEXT_PUBLIC_*` 노출 키 4종** (`APP_URL`·`GOOGLE_MAPS_API_KEY`·`CLOUDINARY_CLOUD_NAME`·`CLOUDINARY_UPLOAD_PRESET`)는 브라우저 번들에 그대로 들어감 — Maps 키는 반드시 referrer 제한, Cloudinary unsigned preset은 업로드 폴더/포맷 제한 필수.
- `lib/env.ts`의 `validateEnv()`는 누락 시 **로그만 남기고 throw 하지 않음** (의도적). 메인 페이지는 DB 미설정에서도 살아있도록 설계됨.
- `.env.production`에는 키 자리표시만 있고 실제 값은 Vercel UI에서 주입.
- Clerk 키 그룹은 코드 흔적은 있으나 활성 사용 여부 모호 — 정책 결정 후 정리 권장.

## 6. 발견된 문제점 Top 10

> 우선순위: **보안 → 성능 → 아키텍처 → UX**. 각 항목은 증거(file:line) 포함.
> ⚠️ 본 보고서에서는 수정하지 않습니다. 후속 PR로 분리 처리 권장.

### 🛡️ 보안

#### 1. (HIGH) Mutation API 라우트 전반에 CSRF 보호 없음
- **위치**: `app/api/**/route.ts` 전반의 POST/PUT/DELETE 핸들러 (~100+개). 미들웨어에서도 토큰 검증 미수행.
- **영향**: 인증된 사용자가 외부 사이트에서 위조 요청에 노출 → 캠페인 변경, 문의 생성, 어드민 액션 위조 위험.
- **방향**: Origin/Referer 검사 + CSRF 토큰(서버 발급, 헤더 검증) 도입. `next-safe-action`이나 자체 미들웨어로 일괄 적용.

#### 2. (HIGH) 블로그 본문이 `dangerouslySetInnerHTML`로 무방비 렌더
- **위치**: `app/[locale]/blog/[slug]/page.tsx:119` — `dangerouslySetInnerHTML={{ __html: html }}`
- **영향**: 마크다운 → HTML 변환 결과를 sanitization 없이 주입. 콘텐츠 소스가 신뢰 가능하더라도 미래 작성자/CMS 연동 시 저장형 XSS 경로.
- **방향**: `rehype-sanitize` 추가 또는 `react-markdown`으로 교체. `dangerouslyAllowHtml: false`.

#### 3. (HIGH) LLM 응답 `JSON.parse` 후 스키마 검증 없음
- **위치**: `app/api/campaign-planner/route.ts:144-145` — 정규식으로 펜스 제거 후 그대로 `JSON.parse`.
- **영향**: 모델이 임의 필드/구조를 반환하거나 프롬프트 인젝션으로 조작될 경우, 다운스트림 코드가 신뢰 불가능한 객체를 사용.
- **방향**: Zod 스키마로 `safeParse`. 실패 시 명확한 에러 응답 + 재시도/폴백.

#### 4. (HIGH) 어드민 대량 작업 라우트에 레이트리밋 미적용
- **위치**: `app/api/admin/bulk-upload/route.ts`, `app/api/admin/media-samples/route.ts` 등 다수. `app/api/inquiry/route.ts:72`은 `withRateLimit` 적용됨 — 일관성 없음.
- **영향**: 어드민 토큰 탈취/내부자 오남용 시 스토리지 폭증·스팸 매체·DoS.
- **방향**: 어드민 mutation은 IP+사용자 단위로 일괄 레이트리밋(예: Upstash Redis). 미들웨어에서 `path.startsWith("/api/admin/")` 패턴으로 적용.

#### 5. (MEDIUM) 어드민 권한 부트스트랩이 환경변수 평문 이메일 리스트
- **위치**: `.env.example:36` — `NEXTAUTH_ADMIN_EMAILS=...`. `lib/email/send-email.ts:28-39`에서 쉼표 분리만 수행.
- **영향**: 어드민 추가/회수 시 재배포 필요, 감사 로그 부재, 키 유출 시 어드민 이메일 노출.
- **방향**: DB의 `User.role` + `AdminTodo`/감사 로그를 단일 진실 소스로. 부트스트랩만 env로 1회 사용 후 무력화.

### ⚡ 성능

#### 6. (HIGH) 어드민 A/B 실험 목록에서 N+1 집계 쿼리
- **위치**: `app/api/admin/ab/experiments/route.ts:27-31` — 실험 N개에 대해 `Promise.all(experiments.map(... aggregateAbExperiment(e.id)))`. 실험당 별도 집계 쿼리.
- **영향**: 실험 수 증가에 따라 어드민 대시보드 응답 시간 선형 증가 → DB 부하.
- **방향**: 단일 SQL(`GROUP BY experimentId`) 또는 마테리얼라이즈드 뷰. 어드민 KPI는 캐시 가능 영역.

#### 7. (MEDIUM) `/api/chat`이 매 요청마다 매체 목록을 DB에서 조회
- **위치**: `app/api/chat/route.ts:38-47` — 컨텍스트 주입용 top 8 매체를 매번 조회, 캐시 없음.
- **영향**: 챗 응답마다 DB 라운드트립 + LLM 호출. 동시 사용자 증가 시 DB 부하·응답 지연.
- **방향**: 메모리/Edge Cache(예: `unstable_cache` TTL 1h) 또는 빌드 시 정적 컨텍스트.

#### 8. (MEDIUM) 어드민 리뷰 폼·뉴스/매체 화면의 비최적 `<img>` 다수
- **위치**: `components/admin/review/media-review-form.tsx`, `components/medias/AdminMediaEditPanel.tsx:422`, `components/medias/AdminCaseStudy*.tsx`, `app/[locale]/medias/[mediaId]/page.tsx:278`, `app/[locale]/news/[slug]/page.tsx:124`, `components/ui/image-carousel.tsx:60`, `app/[locale]/news/news-grid.tsx:111` 등 (`next build` 시 lint 경고).
- **영향**: 대형 원본 이미지가 그대로 로드 → LCP 저하, 모바일 데이터 낭비.
- **방향**: `next/image`로 교체, `sizes`·`priority`·`placeholder="blur"` 설정. 어드민도 동일 적용 (장시간 사용 화면).

### 🧱 아키텍처

### 🎨 UX

#### 10. (MEDIUM) 인증/i18n 흔적 충돌 — Clerk vs NextAuth
- **위치**: `app/[locale]/sign-in`·`/sign-up` (Clerk catch-all 흔적), `app/api/webhooks/clerk/route.ts`, `package.json`의 `svix` — 그러나 실제 인증은 `NextAuth` (`app/[locale]/login`·`/signup`).
